## Abstract

Summarize the proposal in one or two paragraphs.

## Motivation

Explain the problem, user impact, and why a protocol or process change is needed.

## Specification

Describe the proposed behavior precisely enough for independent implementation and review.

## Backward Compatibility

List compatibility impacts for nodes, wallets, miners, APIs, tools, and documentation. Use `None` only when there is no expected impact.

## Security Considerations

Document the security impact of the proposal. At minimum, address:

- Trust boundaries: which users, services, keys, or nodes must trust each other?
- Authentication and authorization: does the proposal add, remove, or change access checks?
- Consensus and settlement safety: can different node versions disagree, fork, double count, or reorder value?
- Wallet and key handling: are private keys, signatures, seeds, nonces, or addresses affected?
- Network and availability risks: can the proposal increase DoS, replay, spam, or rate-limit pressure?
- Data exposure: does the proposal expose miner, wallet, agent, or infrastructure metadata?
- Mitigations and tests: what checks, rollout gates, monitoring, or regression tests reduce the risks above?

If a category does not apply, state why rather than deleting it.

## Reference Implementation

Link to implementation PRs, prototypes, or test fixtures when available.

## Open Questions

List unresolved design or rollout questions.
